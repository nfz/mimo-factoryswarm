# 任务分配与模块严谨定义 (Task Allocation & Detailed Definitions)

为了确保 10 个 Agent 高效协同开发，我们将整个开发过程拆分为 10 个清晰的 Track。本版本对每个模块的核心输入、输出、物理约束（防止幻觉）以及事件流转边界进行了极其严谨的定义。

## Track 01: 核心基建与事件引擎 (The Architect Agent)
**目标:** 这个模块不是单纯的“技术底座”，而是整个产品体验成立的前提。用户之所以会觉得这个项目像一个真正的数字孪生工厂，而不是一堆 Agent 在终端里轮流输出文字，关键就在于这里。它负责把“订单来了”“某工位缺料了”“某台设备坏了”“维修完成了”“插单导致重排产”这些原本分散的动作，组织成一个连续、可信、可回放、可观察的工厂运行过程。换句话说，`Track 01` 决定了这个产品有没有“世界观”，有没有“流水线正在真实运转”的感觉。

从产品视角看，这一层首先要解决的不是代码怎么写，而是用户在 Demo 中到底会看到什么。用户看到的不是一个函数调用链，而是一条产线从接单到出货的动态过程。订单进入系统后，不应该突然跳到“已完成”，而是应该逐步经过排产、入站、加工、质检、返修、包装、出库等阶段。每个阶段都应该在时间线上留下痕迹，形成可追踪的过程记录。这样用户在演示时才会感受到系统不是在“算结果”，而是在“模拟过程”。这对于产品成立非常关键，因为工厂仿真的价值不在于最终只告诉你产量是多少，而在于它能解释产量为什么变高、为什么变低、瓶颈出现在哪里、异常是怎样一路传导成全局延迟的。

因此，`Track 01` 的第一产品职责，是建立一个统一的“仿真时间”。这个时间不是工程实现意义上的 timer，而是产品里的叙事主轴。没有统一时间，所有模块都只是在各说各话，系统也无法向用户展示“第 120 Tick 出现缺料，第 138 Tick AGV 到达，第 160 Tick 恢复加工”这种具有说服力的过程。对用户来说，时间轴就是他理解整个工厂运行逻辑的唯一坐标系。后续不论是实时看板、事件回放、瓶颈分析，还是 benchmark 报告，全部都依赖这条时间轴。如果没有它，产品会退化成一堆静态日志和孤立状态，完全失去“数字孪生”的真实感。

第二个产品职责，是建立统一的“事件语言”。在这个产品里，真正有价值的不是某个 Agent 内部想了什么，而是它对外部世界造成了什么影响。工位缺料、仓库补货、AGV出发、维修接单、质检判废、线长上报瓶颈，这些都应该被抽象成可以被记录、转发、展示和统计的标准事件。产品层面要把这些事件当成系统的“原子叙事单位”。未来我们给用户展示的运行轨迹、给评审展示的 Agent Trace、给 benchmark 展示的异常处理能力，底层都不是凭空编故事，而是由这些事件串起来的。也就是说，事件总线在产品里承担的不是“消息中间件”这么窄的角色，而是承担整个系统的“公共叙事层”。

这意味着 `Track 01` 必须明确界定什么样的信息值得成为事件。不是所有变量变化都要暴露给用户，但凡会影响生产节拍、资源分配、异常升级、订单交付和 KPI 的行为，都应该沉淀为标准事件。例如，工位从空闲变成加工中，这不只是内部状态更新，而是产线进入有效生产的信号；仓库库存跌破阈值，这不只是一个数字减少，而是潜在停线风险的前兆；某工位故障后触发维修流程，这不只是报错，而是一条会传导到全线节拍、良率和交期的异常链路。只有把这些行为标准化，产品才能在上层做出真正可信的可视化和分析。

第三个产品职责，是让系统具备“可解释性”。很多多 Agent 项目最大的问题，是最后只能给出一个结果，却无法说清楚这个结果是怎么来的。用户最容易质疑的点就是：“这到底是认真模拟出来的，还是模型随便编的？” `Track 01` 的价值，就是为整个产品提供一条能够回答这个问题的证据链。因为每一个关键动作都发生在明确的时间点，并且以事件形式被保存下来，所以当用户看到某订单延迟、某产线拥堵、某工位反复返修时，系统可以把原因链路直接拉出来。产品层面上，这会让 Demo 从“看起来很炫”升级为“看起来可信”。这类可信性，恰恰是申请资源、写项目介绍、做展示视频时最值钱的部分。

第四个产品职责，是支撑规模化叙事。我们要讲的是“最多 1000 个轻量 Agent 的工厂协同”，而不是“10 个脚本顺序执行”。如果没有统一时间和统一事件层，这个规模感是无法成立的。因为当 Agent 数量上来之后，用户已经不可能逐个看每个 Agent 的内部推理过程，他只能看系统级流转。所以 `Track 01` 必须天然支持“聚合观察”。例如产品后面可以展示：某一分钟内总共触发了多少物料请求、多少维修事件、多少返修回流、哪些事件导致了瓶颈放大。换句话说，Track 01 不只是底层模块，它其实在提前为整个产品的“大规模观测能力”打基础。

从 Demo 设计来看，这个模块还直接决定画面感。一个好的演示不是展示“这里有一个 event bus”，而是展示“11:20 某条线突然缺料，11:23 AGV 被重新调度，11:30 备用库存补上，11:42 产线恢复”。也就是说，这个模块最终要服务于三个可见成果：第一，用户能够看到系统在往前跑，而不是静态输出；第二，用户能够理解异常是如何发生和传播的；第三，用户能够相信后续的 KPI 和优化建议并非拍脑袋，而是有过程支撑。只要这三点成立，整个 FactorySwarm 项目的产品可信度就会被一下子拉高。

所以，`Track 01` 的产品定义可以总结为一句话：它负责把“多 Agent 的离散动作”组织成“用户能看懂、能追踪、能复盘、能验证”的工厂运行过程。它不是给开发者看的小模块，而是整个产品从“Agent 拼盘”升级为“数字孪生系统”的分水岭。

**本模块在产品中的交付标准:**
*   用户能够看到一条清晰的仿真时间线，而不是一堆无序日志。
*   任意一个关键异常都能被追踪到发生时间、影响范围和后续处理过程。
*   后续模块产出的 KPI、瓶颈分析、异常回放，都能回溯到具体事件链路。
*   Demo 中能够自然演示“订单流转”“异常扩散”“恢复处理”这三类典型过程。

**输出物:**
*   `src/simulator.py`: 负责承载整个仿真流程的推进逻辑，让产品具备连续运行和可回放的时间主轴。
*   `src/event_bus.py`: 负责承载系统中的标准事件流，让不同角色、不同产线、不同异常都能进入统一的产品叙事框架。
*   `src/events.py`: 负责定义产品级事件词典，统一“什么事情值得被系统记录、展示、统计和回放”。

## Track 02: LLM通信与Agent基类 (The Prompt Engineer Agent)
**目标:** 这个模块在产品层面上解决的，不是“怎么调 API”这么狭窄的问题，而是“如何让每一个 Agent 看起来真的像一个有角色、有职责、有行为风格的工厂成员”。如果 `Track 01` 负责建立这个工厂的时间和事件世界，那么 `Track 02` 负责赋予每个 Agent 可被感知的个性、职责边界和响应方式。没有这一层，整个系统虽然可以跑，但用户看到的只会是一堆名称不同、本质相同的自动回复器，完全无法形成“排产员在思考调度、维修员在处理故障、质检员在判定返修、仓库在响应补料”的角色真实感。

从产品视角看，这个模块首先要解决“角色可信度”问题。多 Agent 产品最容易失败的地方，就是虽然定义了很多 Agent 名称，但它们说话方式一样、处理问题方式一样、输出结构也一样，最后用户只会觉得这是一个模型套了十层壳。`Track 02` 的核心价值，就是把不同 Agent 从“同一个模型的不同名字”升级成“产品里能被明显区分的不同岗位”。例如，排产 Agent 的输出应该更偏全局权衡、订单优先级和资源调度；维修 Agent 的输出应该更偏故障判断、修复建议和停机影响；线长 Agent 的输出应该更偏节拍、堆积、瓶颈和风险汇报。也就是说，这一层决定的不是底层推理本身，而是用户到底能不能从结果上感受到“这些角色真的不同”。

第二个产品职责，是把“模型调用”转化成“产品行为”。对用户而言，他并不关心系统背后发了多少次请求，也不关心 SDK 是怎么封装的；他真正关心的是，当工厂里发生一件事时，系统会不会做出符合角色预期的反应。比如某工位缺料了，用户希望先看到工位发出缺料信号，然后仓储和物流开始响应，而不是直接在终端里跳出一句“已处理完成”。这就要求 `Track 02` 不只是负责调模型，而是要定义一套统一的 Agent 行为模式：什么时候应该先观察上下文，什么时候应该给出判断，什么时候应该输出行动，什么时候应该保持沉默。产品上，这相当于给整个多 Agent 系统建立一套“行为语法”，让所有角色的反应都变得可理解、可预测、可调优。

第三个产品职责，是解决“多角色协作的可管理性”。随着 Agent 数量变多，产品最大的风险不是功能不够，而是行为失控。比如某个 Agent 在不该发言时频繁输出，某个 Agent 在应该处理异常时没有响应，某个 Agent 明明只负责局部问题却开始越权给全局调度建议，这些都会让产品显得混乱。`Track 02` 在产品上的意义，就是给每个 Agent 建立边界感。用户需要相信：仓储 Agent 不会替代排产 Agent 做决策，维修 Agent 不会胡乱改订单，质检 Agent 不会越权修改生产节拍。只有当每个角色都守住自己的职责边界，整个产品才会呈现出一种“工厂里每个人都在各司其职”的秩序感。

第四个产品职责，是让 Token 消耗变得“合理且可讲述”。这个项目一个非常重要的外部目标，是证明它为什么天然需要大量模型调用。如果只是简单写一句“系统会调很多次 API”，说服力很弱；但如果产品层能清楚说明：不同 Agent 在不同业务阶段都要进行角色化判断，而且这些判断不是一次性的，而是会随着事件推进不断更新，那么 Token 消耗就有了明确的业务来源。比如排产 Agent 在接单时要规划一次，在插单时要重排一次，在某产线故障时还要再评估一次；质检 Agent 每完成一批产品都要做一次判定；线长 Agent 需要定期输出状态摘要；管理 Agent 需要综合多条线的信息做更高层的决策。这样，Token 消耗不再像“烧钱”，而像是产品在驱动一整个角色网络持续运转。

第五个产品职责，是支撑后续演示中的“Agent 痕迹感”。一个优秀的多 Agent Demo，不能只是告诉用户“任务完成了”，而应该让用户感受到不同角色在这个过程中留下了各自的判断痕迹。比如同样面对一场设备故障，维修 Agent 会说“预计修复时间和备件需求”，线长 Agent 会说“当前堵塞会扩散到哪一段工序”，排产 Agent 会说“是否需要把订单切到另一条线”。这些痕迹不仅让系统更有真实感，也极大增强了 Demo 的可看性。很多评审在看类似项目时，本能会怀疑“是不是其实就一个模型在装多人”，而 `Track 02` 正是在产品层面抵抗这种质疑的关键模块。

因此，这个模块不能只被理解为一个技术适配层，它更像是 FactorySwarm 的“角色引擎”。它负责回答三个产品问题：第一，系统里的每个 Agent 为什么存在；第二，它在什么场景下应该被唤起；第三，它被唤起之后，应该以什么风格和什么权限参与协作。只要这三个问题被回答清楚，后面的各个业务模块才能真正拥有清晰的角色分工，而不是停留在概念层。

站在产品设计的角度，`Track 02` 还应该明确一条原则：用户感知的是“角色”，不是“模型”。也就是说，未来无论底层接的是 MiMo、OpenAI-compatible 接口，还是混合模型路由，产品表层都不应该让用户看到“这是哪个 API 的响应”，而应该始终让用户感觉“这是排产员的决策”“这是维修员的判断”“这是线长的总结”。当这种感知建立起来之后，产品的完成度会大幅提升，因为它不再像一个技术实验，而像一个真正可演示、可理解、可扩展的多角色工厂系统。

**本模块在产品中的交付标准:**
*   用户能够明显区分不同 Agent 的角色、职责和输出风格，而不会觉得它们是同一个模型换名重复发言。
*   每个 Agent 都有清晰的触发场景和行为边界，系统整体表现出“各司其职”的秩序感。
*   Demo 中能够留下明确的角色化痕迹，让评审相信这是一个多角色协同系统，而不是单模型伪装。
*   Token 消耗能够被解释为“角色网络持续运转带来的业务消耗”，而不是无意义的模型调用。

**输出物:**
*   `utils/llm_client.py`: 负责承载模型能力接入，但在产品上更重要的使命是把底层模型能力稳定地映射为可被角色使用的判断能力。
*   `src/agent.py`: 负责定义 FactorySwarm 中 Agent 的统一角色骨架，保证后续每个岗位都能以一致、可管理、可观察的方式参与产品运行。

## Track 03: 物理孪生实体与状态机 (The Physicist Agent)
**目标:** 建立不可违背的物理规则，这层属于“硬代码”，绝不交给 LLM 决定，彻底阻断幻觉。
*   **Entity State Machine:** 
    *   **严谨设定:** 工位状态机只允许以下流转：`IDLE` <-> `WORKING`, `WORKING` -> `WAITING_MATERIAL`, `WORKING` -> `BROKEN`, `BROKEN` -> `MAINTENANCE`, `MAINTENANCE` -> `IDLE`。
    *   **硬逻辑校验:** 如果工位状态是 `BROKEN`，即使 Agent 试图将其改为 `WORKING`，系统底层状态机也必须拦截并抛出异常。
*   **输出物:**
    *   `src/factory.py` (工厂环境单例，管理所有实体的引用)
    *   `src/station.py` (工位数据结构与状态流转拦截器)
    *   `src/order.py` (BOM表、工序路线的数据结构)

## Track 04: 产线与工位智能体 (The Production Dev Agent)
**目标:** 实现一线的生产执行角色，它们负责消耗 Tick。
*   **Station Agent (工位):** 
    *   **严谨设定:** 根据自身负责的工序（如：打螺丝），每个 Tick 扣减加工所需时间（如 `remaining_ticks -= 1`）。加工完成时，发出 `ProcessCompleted` 事件；缺料时，发出 `MaterialRequest` 事件；自身状态机变为 `BROKEN` 时，立刻发出 `MachineFailure` 广播。
*   **Line Supervisor Agent (线长):** 
    *   **严谨设定:** 订阅其管辖的所有 Station Agent 的事件。不负责具体加工，只负责分析瓶颈（例如：工位 B 堆积了 10 个半成品，工位 C 却在 `IDLE`）。向厂长发送 `LineStatusReport`。
*   **输出物:**
    *   `agents/station_agent.py`
    *   `agents/line_supervisor_agent.py`

## Track 05: 仓储与物流智能体 (The Supply Chain Dev Agent)
**目标:** 管理工厂内部的空间物流与库存水位。
*   **Warehouse Agent (仓库):** 
    *   **严谨设定:** 维护一个全局库存字典 `{"screws": 5000, "screens": 200}`。收到 `MaterialRequest` 时扣减库存。如果库存低于阈值，向外部系统（采购）发出 `InventoryWarning`。
*   **Logistics Agent (AGV小车):** 
    *   **严谨设定:** 拥有物理坐标或路径成本属性（如 `current_location`, `target_location`）。从仓库到工位搬运物料需要消耗实际的 Tick（如 `distance / speed`）。搬运途中不能响应其他请求。到达时发出 `MaterialDelivered` 事件。
*   **输出物:**
    *   `agents/warehouse_agent.py`
    *   `agents/logistics_agent.py`

## Track 06: 质检与维修智能体 (The QA & Maintenance Dev Agent)
**目标:** 注入随机性与异常处理流程。
*   **QC Agent (质检):** 
    *   **严谨设定:** 订阅 `ProcessCompleted` 事件。根据设定的良率概率（如 98%）掷骰子。如果判定为次品，拦截进入下一道工序，并将该半成品打回之前的工位（发出 `ReworkRequired` 事件）。
*   **Maintenance Agent (维修):** 
    *   **严谨设定:** 平时处于 `IDLE`。收到 `MachineFailure` 后，改变状态为 `MOVING`（消耗 Tick 到达工位），然后变为 `REPAIRING`（消耗设定的 MTTR，平均修复时间）。修复完成后发出 `RepairCompleted` 事件，并释放该工位。
*   **输出物:**
    *   `agents/qc_agent.py`
    *   `agents/maintenance_agent.py`

## Track 07: 顶层调度与排产智能体 (The Planner Dev Agent)
**目标:** 系统的“大脑”，高度依赖大模型长上下文推理。
*   **Factory Manager Agent (厂长):** 
    *   **严谨设定:** 订阅所有线长的 `LineStatusReport`。根据 OEE（设备综合效率）和产能目标，向 Planner 提出宏观调整要求。
*   **Planner Agent (排产):** 
    *   **严谨设定:** 这是**Token消耗大户**。当收到 `UrgentOrderArrived`（紧急插单）或某产线严重故障事件时，它必须将当前所有产线的状态、进行中的订单、库存数据打包成巨大的 Prompt 交给 LLM。输出结果必须是一个严格的 JSON 格式的重排产计划（如将订单 B 从产线 1 转移到产线 2）。
*   **输出物:**
    *   `agents/planner_agent.py`
    *   `agents/manager_agent.py`

## Track 08: 监控指标与数据埋点 (The Data Analyst Agent)
**目标:** 实时捕获系统的所有动态并提供量化指标，证明该仿真系统的价值。
*   **Metrics Collector:** 
    *   **严谨设定:** 被动订阅 Event Bus 的所有事件。不干扰业务流。
    *   **计算公式:**
        *   `Throughput` (产能) = 单位时间内 `OrderCompleted` 事件的数量。
        *   `Yield Rate` (良率) = 1 - (`ReworkRequired` 次数 / `ProcessCompleted` 次数)。
        *   `OEE` = 稼动率 * 表现性 * 质量指数 (基于各工位非 `IDLE` 且非 `BROKEN` 的 Tick 数计算)。
    *   **Token 账本:** 汇总所有来自 LLM Client 的消耗记录。
*   **输出物:**
    *   `src/metrics.py` (指标计算逻辑与数据导出功能)

## Track 09: 仿真场景编排 (The Scenario Director Agent)
**目标:** 提供工厂运行的初始条件与剧本事件。
*   **Scenarios Generator:** 
    *   **严谨设定:** 提供纯数据配置。定义初始化时实例化多少个 Station, AGV, Line。
    *   **注入事件 (Injectors):** 允许在特定的 Tick 强行向 Event Bus 注入外部事件。例如：`{"tick": 500, "event": "MachineFailure", "target": "Station_B"}`。
*   **输出物:**
    *   `scenarios/baseline_factory.json` (正常运行配置)
    *   `scenarios/shortage_case.json` (物料短缺配置)
    *   `scenarios/machine_failure_case.json` (设备故障剧本)

## Track 10: CLI终端控制台与可视化 (The UI/UX Dev Agent)
**目标:** 将枯燥的后端日志转化为炫酷的控制台动画。
*   **Terminal UI:** 
    *   **严谨设定:** 使用 `Rich` 库实现 `Live` 刷新看板。
    *   **布局要求:**
        *   **Top:** 虚拟时钟 (Tick: 1420), Token 燃烧总数。
        *   **Middle:** 产线看板。用不同颜色的方块代表工位状态 (🟩=工作, 🟨=等待/空闲, 🟥=故障, 🟦=维修中)。
        *   **Bottom:** 实时滚动显示最重要的 Event Bus 日志流 (如 `[ERROR] Station 3 Broken!`, `[INFO] AGV 1 dispatched`)。
*   **输出物:**
    *   `src/main.py` (系统启动入口与 UI 渲染)
